## 2026-07-24 보안 QA — 악의적 사용자 전제 점검
9. **계정 열거 차단 — 구분 정보를 메일함으로 옮긴다**: 가입 요청(`/api/auth/signup/request`) 응답은
   가입 여부와 무관하게 항상 동일하다(기존 409 `already_registered` 제거). 이미 가입된 주소에는
   인증 코드 대신 **"이미 가입된 계정입니다" 안내 메일**을 보낸다.
   → 구분 정보가 HTTP 응답이 아니라 **본인만 볼 수 있는 채널**로만 가므로 UX 손실 없이 열거만 막힌다.
   같은 이유로 OTP 재전송 쿨다운(429)도 삼킨다 — "가입된 주소만 429" 가 되면 그 자체가 오라클이다
   (로그인 경로에도 같은 구멍이 있어 함께 수정). 주소 단위 발송 상한(`mail_to_address`, 5회/시간)으로
   응답 통일 후 생기는 메일 폭탄 경로를 막는다.
10. **CSP `unsafe-inline` 제거는 챗봇(1D) 착수 시점에**: 지금은 앱에 HTML 주입 지점이 0건이고
   (`dangerouslySetInnerHTML` 미사용, React 가 전부 이스케이프) nonce 도입에 미들웨어가 필요해
   비용 대비 실익이 없다. **LLM 응답을 렌더링하는 챗봇이 전제를 바꾸므로** 1D 첫 작업으로 넣는다
   (1D 항목 + DoD 에 명시). 그때까지는 `frame-ancestors 'none'` 등 나머지 지시문이 방어를 담당한다.
11. **세션 무효화 = `users.session_version`**(0010). 별도 세션 테이블을 만들지 않는다.
   JWT 에 발급 시점 세대(`sv`)를 담고 요청마다 DB 값과 대조해 불일치면 401. 값을 1 올리면
   그 계정의 모든 기기가 즉시 로그아웃된다. 권한을 어차피 매 요청 DB 에서 읽으므로(`loadActor`)
   같은 SELECT 에 컬럼 하나를 얹는 것이라 **추가 조회 비용이 없다**. 회원 관리 화면에
   "모든 기기에서 로그아웃" 버튼으로 노출(2회 클릭 확인). sysadmin 대상은 sysadmin 만 가능.
   ※ 이 컬럼은 원래 설계에 없던 것을 이번에 신설한 것이다(0010).
12. **회장단 상호 신뢰 유지 + 안전장치 2개**: sysadmin 만 회장단을 강등하게 만들면 개발자가 졸업 후
   거버넌스의 단일 병목이 된다. 회장단 교체·유고 대응은 회장단이 스스로 할 수 있어야 하므로
   **상호 견제는 코드 차단이 아니라 가시성으로 푼다**.
   (a) **마지막 활성 권한자(회장단·시스템관리자) 1인은 강등·비활성화 불가**(`last_privileged`)
       — 전원 잠금 시 부트스트랩 스크립트 없이는 복구가 불가능하다.
   (b) 회장단 대상 권한 변경·강제 로그아웃은 **audit 에 `[high]` 표기 + 회장단 전원과 공용 메일로 즉시 알림**.
       행위자 본인도 수신자에 포함한다 — 계정 탈취 시 "내가 하지 않은 알림"이 가장 빠른 탐지 신호다.

13. **비활성화·강등 시 세션도 함께 끊는다**(`session_version` + 1): 비활성화의 의도는
   "이 사람의 접근을 **지금** 끊는다" 인데, 세션을 남겨 두면 쓰기만 막히고 화면 열람은 계속돼
   의도와 어긋난다. 운영진을 퇴출하거나 계정 이상이 의심되는 상황에서는 **읽기 접근 자체가 리스크**다.
   강등(회장단→운영진→부원)도 같다 — 역할이 줄었는데 이전 역할로 발급된 토큰이 살아 있을 이유가 없다.
   - 적용: `setMemberActive(false)`, `setMemberRole`(강등일 때만). **재활성화·승격은 올리지 않는다**
     (끊을 이유가 없고, 로그인 중 승격된 사람을 튕겨낼 필요도 없다).
   - 강등 판정은 `isDemotion()` 순수 함수. `board`↔`sysadmin` 은 등급이 같지만 **sysadmin → 그 외**는
     sysadmin 전용 조작을 잃으므로 축소로 본다.
   - audit 의 `after.sessionsRevoked` 로 "이 변경이 세션을 끊었는지" 를 남긴다.
   - 참고: 인가는 원래 매 요청 DB 에서 역할을 다시 읽으므로(`loadActor`) **강등 후 권한이 남는
     구멍은 없었다**. 이 결정은 그 위에 얹는 방어이자 제품 동작의 일관성 확보다.
