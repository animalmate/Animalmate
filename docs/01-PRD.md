# 01. 개발 기획서 (PRD)

## 1. 배경과 문제
- 대상: 약 300명 규모 대학생 동물 봉사 동아리. 소통·기록은 네이버 카페(cafe.naver.com/animalmate2010)와 카카오톡.
- 현재 반복 수작업:
  1) 각 활동팀(1~5팀)이 **월 1~3회, 정해진 요일·시간에 봉사 공지**를 카페에 수동 게시
  2) 신청을 수동 수합(댓글/폼) → 확정자 정리
  3) 확정자 대상 **카톡방을 수동 개설·초대**
  4) 부원들의 반복 질문(회비, 봉사시간 인정, 일정 등)에 운영진이 매번 수동 응대
- 카페의 한계: 기록 보관은 좋으나 "상태"(신청 현황, 정원, 확정 여부)가 구조화되지 않아 매번 사람이 취합.

## 2. 목표
- 팀장단의 공지→수합→카톡방 루프를 자동화/반자동화하여 팀당 월 수 시간의 수작업 제거.
- 부원 반복 질문을 RAG 챗봇이 1차 응대(출처 표시 + 모르면 운영진 핸드오프).
- 운영진(30±10명, 매 학기 교체) 권한을 안전하게 관리: 초대제, 소유권, 임기 자동 만료, 감사 로그.
- 개발자(제작자) 탈퇴 이후에도 운영 가능: 조직 계정, 인수인계 문서, 데이터 export.

## 3. 비목표 (하지 않는 것)
- 네이버 카페를 대체하지 않는다. 사진·영상·후기 아카이브는 카페에 남는다.
- 카페 글 읽기/수정/댓글 자동화는 하지 않는다(API 부재·약관).
- 카톡방 생성/초대 자동화는 하지 않는다(제휴 API 전용). 링크·참여코드 관리만 한다.
- 회계/정산 자동화는 v1 범위에서 제외한다(금전 = 사람 승인 원칙). v2 백로그.
- 부원에게 새로운 습관을 강요하지 않는다. 카페·단톡은 그대로, 새 서비스는 신청·조회·질문만.

## 4. 조직 구조와 역할 (모두 데이터로 관리, 하드코딩 금지)
- 부원: 약 300명. 신청, 후기 제출, 챗봇 질문.
- 운영진: 30±10명, **매 학기 교체**.
  - 활동팀 1~5팀(개수 가변), 각 팀 팀장단 3명(인원 가변)
  - 기능팀: 기획팀, 홍보팀, 총무팀, 대외협력팀 등(가변)
- 회장단(최고 권한): **회장, 부회장, 총무 3인**. 운영진 임명/해제, 학기 전환, 게시판 레지스트리,
  봇 토큰 관리. 회계 등 민감 문서는 회장단 전용 visibility.
- 시스템관리자: 개발자 계정(회장단과 동급 기술 권한). 탈퇴 시 회장단이 회수.

### 권한 요약
| 행위 | 부원 | 운영진 | 회장단/시스템관리자 |
|---|---|---|---|
| 챗봇 질문 / 봉사 신청 | O | O | O |
| 게시물·예약 **생성** | X | O (제한 없음, 어떤 글이든) | O |
| 게시물·문서 **수정/삭제** | X | 소유자(본인/소속팀)만 | O (전체) |
| RAG 문서 편집 | X | 소유 범위 내 | O |
| 운영진 임명/해제, 학기 전환 | X | X | O |
| 게시판 레지스트리, 봇 토큰 | X | X | O |

## 5. 핵심 워크플로 (v1의 심장)
1) **반복 규칙**: 팀별 "매월 N번째 X요일 HH:MM, 대상 게시판, 템플릿" 등록
2) **초안 생성**: 발행 3일 전 크론이 회차 초안 자동 생성 → 담당 팀장단에게 알림
3) **초안 완성**: 팀장단이 일시/장소/정원 등 빈칸 입력 + **오픈채팅방을 미리 만들어
   링크·참여코드 등록** → 확정. 필수 필드 미입력 시 발행 보류 + 재알림
4) **자동 발행**: 정해진 시각에 봇 계정이 카페 글쓰기 API로 해당 게시판(menuid)에 게시.
   공지에는 새 서비스의 신청 링크 포함
5) **신청/확정**: 부원 신청 → 실시간 현황판 → 정원/마감시각 도달 시 자동 마감 →
   선착순 자동 확정 또는 팀장단 선발(팀별 설정)
6) **카톡 안내**: 확정자에게만 신청 확인 페이지/알림으로 오픈채팅 링크+참여코드 노출
7) **상태 조회**: 챗봇이 "이번 주 2팀 봉사 자리 남았어?" 같은 상태형 질문에 실데이터로 답변

## 6. 기능 명세
- F1 반복 공지 엔진: 규칙 CRUD, 회차 초안 생성, 발행 보류 안전장치
- F2 신청/확정: 신청 폼, 현황판, 마감 처리, 확정 방식(선착순/선발), 확정자 전용 정보 노출
- F3 카페 발행 큐: 예약 글쓰기(제목/본문/이미지/게시판/시각), 상태머신, 재시도, 실패 알림,
  발행 결과(카페 URL) 저장. 게시판 레지스트리(menuid 매핑)는 회장단이 콘솔에서 관리
- F4 RAG 챗봇: 문서 기반 답변 + 상태형 질의(이벤트/신청 데이터), 출처 표시, visibility 필터,
  핸드오프, 질문 로그
- F5 문서 관리: 마크다운 CRUD, visibility(member/staff/board), 소유권, 저장 시 자동 재임베딩
- F6 조직/계정: 초대제 가입(이메일 매직링크), 팀/멤버십 관리, **학기 전환**(유임 체크 →
  일괄 만료 → 신규 초대), 임기 자동 만료
- F7 운영 공통: audit log, 봇 토큰 상태 대시보드, 알림(이메일), 데이터 CSV export

## 7. 핵심 설계 결정 (변경 금지 — 근거 포함)
1. 카페 = 콘텐츠 아카이브, 새 서비스 = 상태 층. (미디어 저장 비용 0, 부원 습관 유지)
2. 카페에 쓰는 주체는 **동아리 봇 계정 1개**. 운영진 개인 네이버 계정을 OAuth 연동하지 않는다.
   글 본문 하단에 "작성: ○○팀 ○○" 자동 표기로 실제 작성자를 남긴다.
3. 카페 API는 쓰기 전용·수정 불가 → 갱신되는 정보(신청 현황, 카톡 링크)는 서비스 페이지가 원본.
4. 카톡은 반자동: 방은 사람이 사전 생성, 시스템은 링크+참여코드를 확정자에게만 배포.
5. 보안: 초대제 + 서버측 검증 + 소유권 + 임기 자동 만료 + audit log. 2FA는 회장단·시스템관리자만.
6. 모든 조직 수치(팀 수, 팀장단 수, 운영진 수)는 가변 → 스키마와 UI에 상수 금지.

## 8. 비기능 요구
- 개인정보: 수집 최소화(이름/학번/연락처 중 필요 항목만), 동의 체크, 처리방침 페이지,
  탈퇴 시 삭제. PII는 RAG 인덱스 제외.
- 비용: 무료 티어 중심. 고정비 = 도메인(연 ~2만원) + LLM 크레딧(월 수천 원, 선불·한도 설정).
- 계정: 모든 인프라 계정은 동아리 공용 이메일 소유(별도 자산 대장 문서로 관리).
- 성능: 300명 규모. 챗봇 응답 수 초, 발행 크론 1~5분 주기면 충분.

## 9. 성공 지표 (처음부터 측정)
- 팀장단 회차당 수작업 시간 (도입 전/후)
- 자동 발행 성공 건수 / 실패율
- 챗봇 자동 응답률, 오답률(평가셋 30문항 정기 실행), 핸드오프율
- 주간 활성 운영진 수, 파일럿 팀 → 5팀 확산 여부

## 10. 마일스톤
- Phase 0 (8월 3~4주): 외부 API 검증 + 계정 셋업 → GO/NO-GO 결정
- Phase 1 (9월~10월 중순): 파일럿 — 1개 팀에서 핵심 루프 전체 가동
- Phase 2 (10월 하순~11월): 5개 팀 확산 + 챗봇 고도화 + 지표 대시보드
- Phase 3 (12월~겨울): 학기 전환 실전 투입, 인수인계 문서, v2 백로그 정리
