# 01. 개발 기획서 (PRD)

## 1. 배경과 문제
- 대상: 약 300명 규모 대학생 동물 봉사 동아리. 소통·기록은 네이버 카페(cafe.naver.com/animalmate2010)와 카카오톡.
- 현재 반복 수작업:
  1) 각 활동팀(1~5팀)이 **월 1~3회, 정해진 요일·시간에 봉사 공지**를 카페에 수동 게시
  2) 신청을 수동 수합(댓글/폼) → 확정자 정리
  3) 확정자 대상 **카톡방을 수동 개설·초대**
  4) 부원들의 반복 질문(회비, 봉사시간 인정, 일정 등)에 운영진이 매번 수동 응대
- 카페의 한계: 기록 보관은 좋으나 "상태"(신청 현황, 정원, 확정 여부)가 구조화되지 않아 매번 사람이 취합.

## 2. 목표
- 팀장단의 **정기 봉사 공지 발행**을 자동화하여 팀당 월 수 시간의 수작업 제거(반복 규칙 → 초안
  → 완성 → 자동 발행). 신청 수합·카톡방은 현행(카페 댓글 + 팀장단 수동)을 유지 — 자동화 범위 밖.
- 부원 반복 질문을 RAG 챗봇이 1차 응대(로그인 전용, 출처 표시 + 모르면 운영진 핸드오프).
- 운영진(30±10명, 매 학기 교체) 권한을 안전하게 관리: 학기별 가입코드, 소유권, 임기 자동 만료, 감사 로그.
- 총무 업무(회비 현황·지출 대장)와 신입 모집(안내·결과 조회)을 최소 기능으로 지원.
- 개발자(제작자) 탈퇴 이후에도 운영 가능: 조직 계정, 인수인계 문서, 데이터 export.

## 3. 비목표 (하지 않는 것)
- 네이버 카페를 대체하지 않는다. 사진·영상·후기 아카이브는 카페에 남는다.
- 카페 글 읽기/수정/댓글 자동화는 하지 않는다(API 부재·약관).
- **봉사 신청 수합을 자동화하지 않는다.** 신청은 카페 댓글(현행), 수합·확정은 팀장단 수동.
  시스템은 신청 폼·현황판·확정·정원 관리를 제공하지 않는다(스코프 피벗 2026-07-23).
- 카톡방 생성/초대/참여코드를 시스템이 다루지 않는다(제휴 API 전용·약관). 방은 사람이 관리.
- 자동이체·결제 연동·정산 요청 접수는 하지 않는다(금전 = 사람 승인 원칙). 총무 모듈(F8)은
  회비 납부 현황·지출 기록 대장까지만. 정산 요청은 v2 백로그.
- 부원에게 새로운 습관을 강요하지 않는다. 카페·단톡은 그대로, 새 서비스는 조회·질문 중심.

## 4. 조직 구조와 역할 (모두 데이터로 관리, 하드코딩 금지)
- 부원: 약 300명. 신청, 후기 제출, 챗봇 질문.
- 운영진: 30±10명, **매 학기 교체**.
  - 활동팀 1~5팀(개수 가변), 각 팀 팀장단 3명(인원 가변)
  - 기능팀: 기획팀, 홍보팀, 총무팀, 대외협력팀 등(가변)
- 회장단(최고 권한): **회장, 부회장, 총무 3인**. 운영진 임명/해제, 학기 전환, 게시판 레지스트리,
  봇 토큰 관리. 회계 등 민감 문서는 회장단 전용 visibility.
- 시스템관리자: 개발자 계정(회장단과 동급 기술 권한). 탈퇴 시 회장단이 회수.

## 5. 핵심 워크플로 (v1의 심장 = 반복 공지 발행)
1) **반복 규칙**: 팀별 "매월 N번째 X요일 HH:MM, 대상 게시판, 템플릿" 등록
2) **초안 생성**: 발행 3일 전 크론이 회차 초안 자동 생성 → 담당 팀장단에게 알림
3) **초안 완성**: 팀장단이 일시/장소/정원 등 빈칸 입력 → 확정. 필수 필드(일시/장소/정원)
   미입력 시 발행 보류 + 재알림(빈 공지가 카페에 나가지 않게)
4) **자동 발행**: 정해진 시각에 봇 계정이 카페 글쓰기 API로 해당 게시판(menuid)에 게시.
   **신청은 그 카페 글의 댓글로 접수(현행 유지)** — 수합·확정은 팀장단이 수동으로 처리
5) **상태 조회**: 챗봇(로그인 전용)이 "다음 2팀 봉사 언제야?" 같은 다가오는 봉사 일정 질문에
   events 실데이터로 답변(잔여 인원 등 신청 현황은 제공하지 않음 — 신청은 카페 댓글)

## 6. 기능 명세
- F1 반복 공지 엔진: 규칙 CRUD, 회차 초안 생성, 발행 보류 안전장치
- ~~F2 신청/확정~~ **제거(스코프 피벗 2026-07-23)**: 신청=카페 댓글(현행), 수합=팀장단 수동.
  applications/confirm_mode/openchat 폐기. events 는 공지 발행용 회차 데이터로만 유지.
- F3 카페 발행 큐: 예약 글쓰기(제목/본문/이미지/게시판/시각), 상태머신, 재시도, 실패 알림,
  발행 결과(카페 URL) 저장. 게시판 레지스트리(menuid 매핑)는 회장단이 콘솔에서 관리
- F4 RAG 챗봇: **로그인 사용자 전용**. 문서 기반 답변 + 상태형 질의(events 다가오는 봉사 일정),
  출처 표시, visibility 필터, 핸드오프, 질문 로그. **사용자별 일일 쿼터 + 전역 일일 상한**.
- F5 문서 관리: 마크다운 CRUD, visibility(member/staff/board — public 없음), 소유권, 저장 시 자동 재임베딩
- F6 조직/계정: **학기별 가입코드**로 부원 가입(카페 공지로 코드 배포, 회장단 재발급) + 로그인은
  이메일 매직링크. 팀/멤버십 관리, **학기 전환**(유임 체크 → 일괄 만료 → 새 가입코드 발급),
  임기 자동 만료
- F7 운영 공통: audit log, 봇 토큰 상태 대시보드, 알림(이메일), 데이터 CSV export
- F8 총무(v1 최소, Phase 2): 회비 납부 현황(학기별 부원 명단 대비 미납/납부/면제 체크, 금액·계좌
  미저장) + 지출 기록 대장(일자/분류/내역/금액/영수증 이미지/메모, 승인 플로우 없음).
  **총무·회장단만 접근**(일반 운영진 불가). 정산 요청 접수·자동이체·결제 연동은 제외(v2)
- F9 신입 모집(Phase 2): 비로그인 정적 안내 페이지(LLM 미사용) + 지원 결과 개별 조회
  (이름+전화번호 전체를 해시 대조, 본인 상태만; 전화 원문 미저장). 상태 5단계(접수→서류합격→
  면접예정→최종합격|불합격). 지원자 명단은 운영진 CSV 업로드, 모집 종료 시 cohort 일괄 삭제
  (비부원 개인정보 보관 금지). 조회 남용 방지(IP 레이트리밋·차단)

## 7. 핵심 설계 결정 (변경 금지 — 근거 포함)
> 2026-07-23 스코프 피벗으로 회장단 지시하에 개정(3·4·5 수정, 7·8 추가). 이후 재변경은 다시 금지.
1. 카페 = 콘텐츠 아카이브, 새 서비스 = 상태 층. (미디어 저장 비용 0, 부원 습관 유지)
2. 카페에 쓰는 주체는 **동아리 봇 계정 1개**. 운영진 개인 네이버 계정을 OAuth 연동하지 않는다.
   글 본문 하단에 "작성: ○○팀 ○○" 자동 표기로 실제 작성자를 남긴다.
3. 카페 API는 쓰기 전용·수정 불가 → 갱신되는 정보(봉사 일정 등)는 서비스 페이지·챗봇이 원본.
4. **봉사 신청·확정·카톡방은 시스템이 다루지 않는다.** 신청=카페 댓글(현행), 수합·확정=팀장단
   수동, 카톡방=사람이 관리. (제휴 API 전용·약관 + 부원 습관 유지. 피벗 2026-07-23)
5. 보안: **학기별 가입코드** + 로그인 매직링크 + 서버측 검증 + 소유권 + 임기 자동 만료 + audit
   log. 2FA는 회장단·시스템관리자만.
6. 모든 조직 수치(팀 수, 팀장단 수, 운영진 수)는 가변 → 스키마와 UI에 상수 금지.
7. **챗봇은 로그인 사용자 전용**(비로그인 public 공개·visibility public 단계 없음). 근거: LLM 비용
   폭주 방지(사용자별·전역 일일 상한) + visibility 필터로 역할별 정보 보호 + PII/인젝션 노출면 축소.
8. **신입 모집 지원자(비부원)의 PII 최소화**: 결과 조회는 이름+전화번호 전체를 **해시 대조**
   (전화 원문 미저장), 저장은 최소 항목, **모집 종료 시 일괄 삭제**. 조회 남용 방지(IP 레이트리밋·차단).
   지원자 데이터는 RAG 인덱스에 넣지 않는다(규칙 #5 연장).

## 8. 비기능 요구
- 개인정보: 수집 최소화(이름/학번/연락처 중 필요 항목만), 동의 체크, 처리방침 페이지,
  탈퇴 시 삭제. PII는 RAG 인덱스 제외. **신입 모집 지원자(비부원)는 최소 항목만 저장하고
  모집 종료 시 일괄 삭제**(전화번호 원문 미저장 — 이름+전화 전체의 해시만 보관).
- 비용: 무료 티어 중심. 고정비 = 도메인(연 ~2만원) + LLM 크레딧(월 수천 원, 선불·한도 설정).
- 계정: 모든 인프라 계정은 동아리 공용 이메일 소유(별도 자산 대장 문서로 관리).
- 성능: 300명 규모. 챗봇 응답 수 초, 발행 크론 1~5분 주기면 충분.

## 9. 성공 지표 (처음부터 측정)
- 팀장단 회차당 수작업 시간 (도입 전/후)
- 자동 발행 성공 건수 / 실패율
- 챗봇 자동 응답률, 오답률(평가셋 30문항 정기 실행), 핸드오프율
- 주간 활성 운영진 수, 파일럿 팀 → 5팀 확산 여부

## 10. 마일스톤
- Phase 0 (8월 3~4주): 외부 API 검증 + 계정 셋업 → GO/NO-GO 결정
- Phase 1 (9월~10월 중순): 파일럿 — 1개 팀에서 핵심 루프 전체 가동
- Phase 2 (10월 하순~11월): 5개 팀 확산 + 챗봇 고도화 + 지표 대시보드
- Phase 3 (12월~겨울): 학기 전환 실전 투입, 인수인계 문서, v2 백로그 정리
